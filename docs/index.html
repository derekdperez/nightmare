Install Nightmare

There are two sane ways to start: a local Python setup for development and exploration, or a Docker-centered deployment path if you want the coordinator and worker model.

Local Python setup

git clone https://github.com/derekdperez/nightmare.git
cd nightmare

python -m venv .venv
# macOS / Linux
source .venv/bin/activate
# Windows PowerShell
# .\.venv\Scripts\Activate.ps1

pip install --upgrade pip
pip install -r requirements.txt

This is the best path for reading code, testing the crawler locally, iterating on plugins, and validating configuration changes before you involve Docker or cloud infrastructure.

Docker-based setup

docker build -t nightmare:latest .

cd deploy
docker compose -f docker-compose.central.yml --env-file .env up -d --build

The deployment docs describe a central VM running the coordinator server plus Postgres, with worker VMs running the coordinator-side processing loop. That makes Docker the natural choice for production-style use.

First-run checklist

  1. Clone the repository.
  2. Install Python dependencies from requirements.txt.
  3. Review the config/ directory before running anything live.
  4. Set environment values such as API keys or coordinator tokens as needed.
  5. Start with a small target and disable optional features you do not need yet.

Useful entry points

# local crawl
python nightmare.py https://example.com --standalone

# batch processing
python nightmare.py --batch-workers 4

# register targets with central coordinator
python register_targets.py \
  --server-base-url https://YOUR-COORDINATOR \
  --api-token YOUR_TOKEN \
  --targets-file targets.txt

Frequently asked questions

These answers are shaped around the repository docs and code layout, with special attention to deployment, performance, and operating style.

Is Nightmare only a crawler?

No. The repository shows a broader pipeline. Nightmare handles crawling and inventory; Fozzy handles permutations and fuzzing; extractor and reporting components appear to process and summarize results; and coordinator/server tooling manages multi-machine workflows.

Can I run it locally without the coordinator?

Yes. The main crawler script documents standalone local usage. That makes local development and experimentation feasible without bringing up the central server or worker fleet.

Does the HTTP queue slow things down?

A bit, according to the performance doc. The queue adds persistence, polling, retries, and durability overhead for certain HTTP paths. The doc also says the main Scrapy crawl itself does not use that queue, so crawl speed depends more on delays, page limits, targets, and network behavior.

What is the recommended production shape?

The deployment guide points to a central EC2 VM for the server and Postgres, with any number of worker VMs claiming tasks from the coordinator. Docker Compose is used for both central and worker stacks.

Where should I put this website for GitHub Pages?

Put the file in docs/index.html and configure GitHub Pages to publish from the docs/ folder on the main branch. That also matches the latest repository commit shown in the history view.

What if the blog cannot load commits?

The site uses the public GitHub API. If GitHub rate limits the browser or the network is unavailable, the rest of the site still works, and the blog section shows a fallback message rather than breaking the whole page.

Create your own plugins

The repository already has a plugins/ directory, which is the clearest signal that extension points are expected. Even before reading every implementation detail, there is a clean way to structure plugin work so it stays maintainable.

Plugin design goals

  • Keep the plugin focused on one capability.
  • Define stable input and output contracts.
  • Prefer pure functions over hidden global side effects.
  • Make failure modes explicit and logged.
  • Write outputs to predictable artifact locations.
  • Keep plugin config isolated from core config where possible.

Suggested plugin workflow

  1. Create a new module under plugins/.
  2. Accept normalized crawl or fuzz artifacts as input.
  3. Transform or enrich them in a deterministic way.
  4. Write structured JSON or HTML output into the domain’s output tree.
  5. Register or invoke the plugin from the relevant stage entry point.

Starter plugin skeleton

"""
plugins/example_plugin.py
A minimal plugin shape you can adapt to Nightmare workflows.
"""

from __future__ import annotations

import json
from pathlib import Path
from typing import Any


def plugin_name() -> str:
    return "example_plugin"


def run_plugin(input_payload: dict[str, Any], output_dir: str | Path) -> dict[str, Any]:
    output_path = Path(output_dir).expanduser().resolve()
    output_path.mkdir(parents=True, exist_ok=True)

    findings = []
    urls = input_payload.get("urls", [])

    for url in urls:
      findings.append({
        "url": url,
        "label": "observed",
        "notes": "This record was produced by the example plugin."
      })

    result = {
      "plugin": plugin_name(),
      "input_count": len(urls),
      "findings": findings,
    }

    report_file = output_path / f"{plugin_name()}.json"
    report_file.write_text(json.dumps(result, indent=2), encoding="utf-8")
    return result


if __name__ == "__main__":
    sample = {
      "urls": [
        "https://example.com/",
        "https://example.com/login",
      ]
    }
    out = run_plugin(sample, "./output/plugin-demo")
    print(json.dumps(out, indent=2))

How to hook plugins into the app cleanly

The safest pattern is to let core stages produce normalized artifacts first, then let plugins consume those artifacts after the fact. That keeps the crawler, coordinator, and fuzzing logic understandable while still giving you room to extend reporting, enrichment, filtering, tagging, or export pipelines.

A good plugin should be easy to remove. If deleting it would force a rewrite of the core crawler or coordinator, it probably grew into the wrong abstraction boundary.
