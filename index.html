Install generate_income.sh

The safest way to start is local Python installation. Once that works, move toward the Docker and coordinator workflow if you want a more operational deployment.

Local development setup

git clone https://github.com/derekdperez/generate_income.sh.git
cd generate_income.sh

python -m venv .venv
# macOS / Linux
source .venv/bin/activate
# Windows PowerShell
# .\.venv\Scripts\Activate.ps1

pip install --upgrade pip
pip install -r requirements.txt

This path is ideal for reading the code, validating crawler behavior, testing local runs, and iterating on extensions without touching cloud infrastructure first.

Docker-oriented setup

docker build -t generate-income:latest .

# if your deployment files use compose
cd deploy
docker compose up -d --build

Use this when you want the broader coordinator and worker model, or when you want repeatable environments across multiple machines.

First-run checklist

  1. Clone the repository and install dependencies.
  2. Review the files in config/.
  3. Inspect target, queue, and API-related settings before a real run.
  4. Start with a small target and low concurrency.
  5. Review output artifacts before scaling up.

Frequently asked questions

These answers are written for someone trying to understand the repo structure, local workflow, and likely deployment shape.

Is this just one shell script?

No. Despite the playful name, the repository is a larger Python-heavy project with dedicated apps for Nightmare, Fozzy, coordinator, server, reporting, plugins, tests, and deployment.

Can I use it locally first?

Yes. The layout strongly supports a local development path first, then a broader coordinator and worker model later if you want to scale or centralize execution.

Why call it generate_income.sh?

It reads like intentional humor layered over a serious codebase. The name is playful, but the structure underneath looks like a real multi-stage platform.

Why might GitHub Pages not show the site yet?

The most common reason is that Pages is not set to deploy from main /(root), or the deployment has not completed yet. A root index.html is necessary, but it is not enough unless Pages is enabled for the repository.

Will the blog work if GitHub’s API is unavailable?

The rest of the site still works. The commit blog progressively enhances itself with live commit data, but the core pages do not depend on that fetch.

Create your own plugins

The repository has a dedicated plugins/ directory, which is a strong hint that customization is part of the intended architecture rather than an afterthought.

Good plugin habits

  • Do one thing well.
  • Take structured input and return structured output.
  • Write artifacts into predictable paths.
  • Log failures explicitly.
  • Avoid hidden global state.
  • Stay easy to delete and replace.

Suggested flow

  1. Create a module under plugins/.
  2. Accept normalized crawl or fuzz artifacts.
  3. Transform, enrich, or score them.
  4. Write JSON or HTML output to the output tree.
  5. Invoke it from the relevant stage or reporting pass.

Starter plugin example

"""
plugins/example_plugin.py
Minimal plugin skeleton for generate_income.sh
"""

from __future__ import annotations

import json
from pathlib import Path
from typing import Any


def plugin_name() -> str:
    return "example_plugin"


def run_plugin(input_payload: dict[str, Any], output_dir: str | Path) -> dict[str, Any]:
    output_path = Path(output_dir).expanduser().resolve()
    output_path.mkdir(parents=True, exist_ok=True)

    findings: list[dict[str, Any]] = []
    urls = input_payload.get("urls", [])

    for url in urls:
        findings.append({
            "url": url,
            "label": "observed",
            "notes": "Generated by example_plugin."
        })

    result = {
        "plugin": plugin_name(),
        "input_count": len(urls),
        "findings": findings,
    }

    (output_path / f"{plugin_name()}.json").write_text(
        json.dumps(result, indent=2),
        encoding="utf-8",
    )
    return result


if __name__ == "__main__":
    sample = {
        "urls": [
            "https://example.com/",
            "https://example.com/login"
        ]
    }
    print(json.dumps(run_plugin(sample, "./output/plugin-demo"), indent=2))
